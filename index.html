Nextflow｜資料是怎麼透過 Channels 在 Processes 之間傳輸？

發表於 2023-01-05 分類於 bioinformatics 閱讀次數： Disqus： 文章字數： 628 所需閱讀時間 ≈ 2 分鐘

Process 是 nextflow 管理分析流程的基本單位，

Process 之間彼此獨立，各有各的工作目錄，也可以分別設定其執行環境（例如 docker container 或 conda environment）。Channel 則媒介了 process 間的資料交流，舉凡字串、數值、檔案路徑乃至標準輸出等資料，皆有對應的 channel 可以使用。若不透過 channels 串接，process 間的檔案或變項無法共享。

假設今有 A 與 B 兩個 process，欲將 A process 定義的變項與寫入的檔案傳遞給 B process 使用，可考慮以下途徑。
查看兩者的工作目錄可以發現，雖然都是字串，但會用不同形式存放在不同位置

在以下案例中，process A 印出的兩條字串以不同的方式存放與傳遞：
string 1 存在檔案裡 (str.txt)，導出為檔案路徑 (path)，透過 queue channel 傳遞給 process B。在這種情況，nextflow 會把檔案路徑用軟連結 (symbolic link) 掛到 process B 的工作目錄，這樣 process B 就能讀取 process A 寫入的檔案。

至於 string 2 則在 process A 裡被定義為變項 (strAsEnv)，再導出為 nextflow 環境變項 (env)，透過 value channel 傳遞給 process B。這種情況則是 bash 變項被導出為 nextflow 變項，存放於 process A 工作目錄底下的 .command.env，隨後輸入 process B 給它印出來，沒有經過存檔與讀檔的過程。

Nextflow 在執行流程時，會為每個 process 建立暫時的工作目錄（預設位置為執行路徑下的work），直接觀察 process A 與 B 能更清楚兩種資料傳遞方式的差異1。

Vim｜Vimium，學習 Vim 的新途徑

發表於 2022-12-31 更新於 2023-01-04 分類於 programming ， tools 閱讀次數： Disqus： 文章字數： 2.2k 所需閱讀時間 ≈ 8 分鐘

Vimium 是 Chrome 的一款擴充套件，它借鑒了 Vim 的鍵位設計與操作邏輯，讓用戶只需要鍵盤便能執行分頁管理、連結點擊與頁面瀏覽等操作。

由於瀏覽器終究不等同編輯器，所以 Vimium 並沒有移植 Vim 所有的功能與指令。然而，即使只納入 Vim 的部分性質，Vimium 仍顯著改善了 Chrome 的鍵盤瀏覽體驗。以點擊連結為例，Chrome 的預設作法是慢慢用 Tab 切到連結位置；Vimium 則會自動標記視窗內的連結，用戶只要鍵入標記字符即可開啟連結。因為最多輸入三個字即可開啟連結，所以有時甚至比滑鼠點擊還迅速。

對於習慣鍵鼠操作卻有心學習 Vim 的用戶而言，一時要以 Vim 取代既有的工作流程，可能是繁瑣且費時的過程。若先從 Vimium 著手，在體驗鍵盤瀏覽網頁的流暢感之餘，也能讓 Vim 操作的抽象性質融入日常生活（例如：查資料與逛論壇）。待初探了 Vim 的設計緣由，再逐步精進其他操作模式，也不失為一種學習 Vim 的可行策略。
